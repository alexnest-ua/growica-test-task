# Рішення, нотатки та компроміси

Обґрунтування вибору реалізації — частина «як я думаю».

## Чому GeneratePress як батьківська тема

Обрано замість Astra й Kadence, бо це **найлегша** з трьох (~30 КБ, мінімальна
типова розмітка), а її **архітектура хуків/фільтрів** дозволяє двом дочірнім
темам розходитися **структурно в коді** — позиція навігації, хедер і футер — майже
без дублювання шаблонів:

- `generate_navigation_location` пересуває меню (під хедером проти плаваючого
  справа) одним фільтром;
- дії `generate_footer` можна зняти й замінити, тож кожен футер — власна
  структура, визначена в дочірній темі;
- мінімальні, передбачувані назви класів роблять перевизначення CSS чистими.

Astra й Kadence важчі й несуть більший, упізнаваніший типовий слід — іронічно для
вправи з усунення сліду. GeneratePress дозволяє зробити розходження **адитивним і
свідомим** — саме та навичка, яку перевіряє завдання.

## Чесне напруження: одна батьківська тема на дві дочірні

Використання **однієї** батьківської теми для обох дочірніх саме по собі є
спільним слідом (ті самі базові назви класів, та сама структура ресурсів). Бриф
цього вимагає, а суть вправи — показати, що можна зробити дві дочірні теми
сприйнятними як різні продукти. У реальному усуненні сліду в масштабі я б також
**варіював батьківську тему між сайтами** (напр. GeneratePress, _s/underscores,
власноруч зроблена тема), варіював набір плагінів і розходився на рівні
інфраструктури — див. [документ рішення](02-solution.uk.md). Це зазначено, щоб
обмеження було явним, а не прихованим.

## Два продукти, а не дві «шкірки»

Розходження керується наданням кожній темі окремої *продуктової персони*, щоб
колір, типографіка й структура розходилися природно, а не довільно:

- **Verdal** — редакційна / wellness: теплий зелений, serif-заголовок (Lora),
  центрований masthead, спокійний ритм читання, світлий 3-колонковий футер.
- **Meridian Edge** — продукт / інженерія: холодний кобальт + чорнило,
  grotesque-заголовок (Space Grotesk), розділений хедер лого-зліва/меню-справа,
  темний 4-колонковий футер.

Візуальну шліфовку свідомо тримали скромною — бриф явно цінує функціональну й
структурну відмінність понад красу.

## Хуки замість перевизначення шаблонів

Файли шаблонів батьківської теми не копіювалися. Структура хедера/футера/
навігації досягається хуками GeneratePress + CSS. Це тримає кожну дочірню тему
мінімальною, уникає розбіжностей із розміткою батьківської й зберігає власні хуки
батьківської для плагінів. (DRY / YAGNI: перевизначати лише потрібне.)

## ACF: Local JSON, лише безкоштовні поля

- **Local JSON** (закомічено в `acf-json/`, авто-завантаження через
  `acf/settings/load_json`) — рекомендований ACF відтворюваний робочий процес і
  саме те, що просить бриф («відтворюваність із репозиторію, а не лише кліками»).
- Ключі полів **читабельні** (`field_verdal_intro_eyebrow`), а не випадкові, тож
  JSON можна оглянути в PR.
- Використано лише **безкоштовні** типи полів ACF (text, textarea, link, image,
  true/false, select) — без Repeater/Flexible Content із Pro.

## Шрифти через Google Fonts CDN

Для хостованого демо CDN Google Fonts із `display=swap` + `preconnect` — це
прагматичний вибір. Для продакшну я б **самостійно хостив** woff2-підмножину
(приватність / GDPR, на одне стороннє з'єднання менше, кращий LCP) — зазначено як
наступний крок, тут не зроблено, щоб лишитися в межах обсягу.

## Помилки, виявлені під час перевірки (і виправлені)

Тестування в реальному браузері виправдало себе:

1. **Кредит GeneratePress усе ще рендерився.** Виклик `remove_action()` дочірньої
   теми для футера GP виконувався під час парсингу, *перед* тим, як батьківська
   зареєструвала ці колбеки (`functions.php` дитини завантажується першим).
   Виправлення: відкласти видалення до `after_setup_theme` в обох темах. Якби
   лишити — це був би і дубльований копірайт, і спільний слід.
2. **Третя колонка футера Verdal вийшла за контейнер.** `<nav>`, відкритий у
   футері, було закрито `</div>`, тож браузер закрив grid зарано. Виправлення:
   закрити `</nav>`. Виявлено порівнянням живого DOM із HTML від сервера.

Обидва записані в історії git як коміт `fix:`, а не «зачищені», тож цикл
перевірка→виправлення видно.

## Ризики та мінімізація (ця реалізація)

| Ризик | Мінімізація |
|-------|-------------|
| Безкоштовні хостовані демо закінчуються | Використати InstaWP (стабільніше); README зазначає, що посилання можуть потребувати оновлення; теми все одно встановлюються з репозиторію. |
| Налаштування Customizer GP не в репозиторії | Усі структурні рішення — у **коді**, тож тема відбудовується без жодного стану Customizer у БД. |
| Самопідписаний сертифікат на локальній розробці | Перевірено через `https://testwp`; продакшн працює з валідним TLS. |
| Колізія назви/slug теми в межах мережі | Кожен сайт використовує унікальні Theme Name, текстовий домен і префікси; у масштабі — генерувати ідентичності для кожного сайту. |

## Інші нотатки

- Бриф завдання у **git-ignore**, а не опублікований — це приватний
  співбесідний матеріал компанії, а репозиторій публічний.
- `noindex` застосовується як **налаштування сайту** (Settings → Reading) на
  кожному демо, а не зашито в темі — індексація це питання розгортання, а не теми.
- Коміти шарові й конвенційні; ідентичність git встановлено **локально** на
  `alexnest-ua`, щоб не використовувати типовий акаунт машини.
